Stückzählung

Funktionsweise

Mit bekannter Anzahl gleicher Teile kalibrieren. Die App rechnet das Stückgewicht aus und zeigt für jedes neue Gewicht die Anzahl.

Beispiel Werkstatt

10 Schrauben auflegen, 10 eingeben, kalibrieren. Beim weiteren Auflegen sieht man live die Stückzahl — schneller als jedes Zählen von Hand.

Beispiel Apotheke

50 Tabletten als Referenz, kalibrieren. Beim Befüllen einer Bestellung zeigt die App, wie viele Tabletten in der Schale liegen.

Genauigkeit

Mehr Referenzteile = genauer. Bei Teilen unter 1 g mindestens 20 Stück, sonst kippt die Anzeige wegen der Auflösungsgrenze 0,1 g.

5 — Architektur: Scale-Modul vs. App-Modul (klar getrennt)
Reine Waagen-Funktion eigenständig — App-Komfort baut nur auf

Das Scale-Modul kann eigenständig laufen — auslesen, steuern, Stream. Drittsysteme (eigene Apps, Workflows, Skripte) können es so direkt ansprechen, ohne den UI-Komfort-Teil mitziehen zu müssen. Das App-Modul ist eine Erweiterung, die auf dem Scale-Modul aufbaut und Toleranz, Netto, Zählen und Erfassen liefert.

Scale-Modul

Reine Hardware-Funktion · /scale/*

Macht nur das Auslesen und Steuern der Waage. Liefert aktuelle Werte über REST und WebSocket, akzeptiert Steuerbefehle (Tara, Einheit, Beleuchtung). Funktioniert ohne das App-Modul.

GET/scale/weightletzter Wert
GET/scale/weight/stablewartet bis stabil
GET/scale/historyRingpuffer
WS/scale/streamLive-Push
POST/scale/command/tareAuf Null setzen
POST/scale/command/unitMaßeinheit
POST/scale/command/lightBeleuchtung
GET/scale/healthReader-Status
GET/scale/modelsbekannte Modelle
GET/PUT/scale/configaktives Modell
Eigenständig deploybar. Andere Anwendungen können nur auf /scale/* zugreifen.

App-Modul

UI-Komfort-Features · /app/*

Setzt das Scale-Modul voraus. Bietet die Werkzeuge, die unsere Web-Anwendung zusätzlich liefert: Toleranz-Ampel, Netto-Tara, Stückzählung, Erfassen mit CSV-Export, Differenz-Wiegen.

GET/POST/DEL/app/toleranceQC-Ampel
GET/POST/DEL/app/nettoBrutto/Tara/Netto
GET/POST/app/countStückzählung
GET/POST/DEL/app/samplesMess-Snapshots
GET/app/samples/statsStatistik
GET/app/samples/export.csvCSV-Download
GET/POST/DEL/app/differenzMehrfach-Tara
GET/app/messlogDiff-Liste
Optionale Erweiterung. Liest passiv vom Scale-Stream, baut den UI-State auf. 
   ┌──────────────────┐
   │  Hardware-Waage  │  RS232 / FTDI / Nullmodem
   └────────┬─────────┘
            │
   ┌────────▼─────────┐
   │   Scale-Modul    │   eigenständig deploybar
   │   /scale/*       │
   │   - Reader/Parser│
   │   - WS-Stream    │
   │   - Befehle      │
   └────┬───────┬─────┘
        │       │
        │       └────────► Drittsysteme (z.B. SPS, eigene Skripte)
        │
   ┌────▼─────────────┐
   │   App-Modul      │   setzt Scale voraus
   │   /app/*         │
   │   - Tolerance    │
   │   - Netto        │
   │   - Count        │
   │   - Samples      │
   │   - Differenz    │
   │   - Messlog      │
   └────┬─────────────┘
        │
   ┌────▼─────────────┐
   │   Web-Frontend   │   Svelte-UI
   │   ScaleApi-Klasse│ (nutzt nur /scale/*)
   │   AppApi-Klasse  │ (nutzt nur /app/*)
   │   LiveStore      │ (vom Stream gefüttert)
   └──────────────────┘

Konsequenz: Wenn jemand die Waage nur in seinen Workflow integrieren will (z.B. eine Versandsoftware fragt das aktuelle Gewicht ab), muss er nur das Scale-Modul kennen. Die UI-Erweiterungen (Toleranz, Zählen, …) sind komplett optional und stören nicht. Auch Updates an einem der beiden Module bleiben jeweils im eigenen Bereich.
